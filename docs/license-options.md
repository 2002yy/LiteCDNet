# LICENSE 选择建议

这个文档的目标不是替你自动选择最终许可证，而是帮你在公开 LiteCDNet 仓库时做一个更稳妥的决定。

## 1. 先说结论

如果你的目标是：

- 让老师、同学、外部读者容易查看和复现
- 允许别人基于你的整理版仓库继续研究或二次开发
- 不强制别人把后续衍生代码继续开源

那么最推荐的选择是：

- `MIT`

如果你的目标是：

- 允许使用
- 允许修改
- 但希望衍生项目在分发时也继续开源

那么更接近你诉求的是：

- `GPL-3.0-only` 或 `GPL-3.0-or-later`

如果你的目标只是：

- 先把代码放到 GitHub 展示
- 暂时不准备授权别人复用、再发布或商用

那就可以：

- 暂时不放 `LICENSE`

这意味着默认仍然是“保留所有权利”，别人可以看，但默认没有被授予复用权。

## 2. 对当前仓库最现实的三种选择

### 方案 A：MIT

优点：

- 最简单
- 最常见
- 对公开科研代码和学生项目最友好
- 别人复现、引用、改造的法律门槛最低

代价：

- 别人可以在保留版权声明的前提下继续使用、修改、再分发
- 也可以把衍生内容放进更闭源的项目里

适合你如果：

- 你更在意传播、展示和复现便利
- 你希望这个仓库“看起来像一个标准公开研究仓库”

### 方案 B：GPL-3.0

优点：

- 同样允许使用、修改和分发
- 但分发衍生版本时通常也要继续开源
- 对“希望后续改动不要被完全闭源拿走”更友好

代价：

- 合规要求更强
- 一些人会因为 copyleft 约束而不愿意直接集成

适合你如果：

- 你明确希望衍生版本继续开源
- 你接受比 MIT 更强的许可约束

### 方案 C：暂不放 LICENSE

优点：

- 不会因为匆忙选错而放出超出预期的授权
- 适合“先公开展示，再决定授权边界”

代价：

- 别人虽然能看到代码，但默认没有明确的复用许可
- 对外部协作者和复现者不够友好
- 仓库的开源感和可复用性会明显下降

适合你如果：

- 你还没想清楚授权边界
- 你更关心展示而不是开放复用

## 3. 结合当前仓库现状的建议

LiteCDNet 当前仓库有几个特点：

- 包含项目内核心实现
- 也包含多个对比模型的项目内适配版本
- 已经补了来源说明、文件级引用映射和 `NOTICE`
- 公开目标更偏向“研究复现仓库”，不是闭源产品仓库

同时还要注意：

- 顶层 `LICENSE` 只能说明你对“这个仓库中你有权授权的整理内容”采取什么授权方式
- 对于 `src/compare/` 中基于论文或公开仓库整理进来的文件，仍然应继续保留来源说明，并尊重其上游许可与引用要求
- 也就是说，顶层 `LICENSE` 不是替代 `docs/attribution.md`、`docs/references.md` 和 `NOTICE.md` 的

基于这些特点，我给你的排序建议是：

1. `MIT`
2. `GPL-3.0`
3. 暂不放 `LICENSE`

也就是说，如果你没有特别强的 copyleft 诉求，`MIT` 会是最省心、也最符合这个仓库气质的方案。

## 4. 不建议现在选什么

不建议你现在直接选下面这些，除非你很确定：

- `Apache-2.0`
理由：它本身没有问题，但相对 MIT 更长、更正式，通常在你需要更明确的专利条款时才更有意义。对一个本科毕设公开版仓库来说，收益未必比复杂度高。

- 自定义“仅学习用途 / 禁止商用”许可证
理由：这种写法最容易与第三方代码来源、未来复用边界和 GitHub 用户预期产生冲突，也不如标准许可证清晰。

## 5. 我建议你怎么做

如果你想今天就把事情做完整：

- 直接选 `MIT`

如果你还想保守一点：

- 先保持现状
- 等你确认“是否允许别人复用和再发布”后，我再帮你补正式 `LICENSE`
